# Proof roadmap

This is the current public proof surface after the top-down virtio proof repair. It is intentionally small. The existence of these modules means Microvmm now has a browseable proof tree, not that the whole VMM is formally verified.

This is also the proof surface that currently builds under `lake build Microvmm.Proof`. During that repair, stale runtime/MMIO whole-loop equalities were intentionally dropped; the current runtime-facing theorems stop at MMIO-exit witnesses, recovered executor reports, and the explicit raw-boundary assumptions they still depend on.

For the reader-oriented overview, start with [proof-engineer-guide](proof-engineer-guide.md). For subsystem structure, see [architecture](architecture.md) and [module-map](module-map.md).

## What exists today

| Proof module | What it states now | Why it matters |
| --- | --- | --- |
| `Microvmm/Proof/Serial/Replay.lean` | The replay buffer stays within its configured tail bound, and the list view of replay bytes has the same length as the stored byte array. | Late-attach console reasoning can talk about a bounded replay buffer without reopening byte-array implementation details. |
| `Microvmm/Proof/Host/ConsoleRuntime.lean` | Successful output queueing keeps each client backlog within the configured cap, and attaching a client adds exactly one runtime client entry. | The host console server has named boundedness and structural-growth facts instead of relying only on code inspection. |
| `Microvmm/Proof/Kvm/Resource.lean` | The KVM guest-read, guest-write, MMIO-observer, IRQ, and run-entry wrappers used by the virtio path are transparent one-step decoders over the raw FFI calls. | The remaining trusted boundary on the proved virtio path is now stated explicitly at the raw shim/Lean runtime edge rather than at an intermediate Lean wrapper layer. |
| `Microvmm/Proof/Kvm/RawBoundary.lean` | The raw FFI contract still trusted by the proved virtio `processQueue` path is packaged explicitly as the observed MMIO/run-entry boundary, guest-memory read/write boundary, and Linux IRQ boundary used by those theorems. | The trusted base is now named as a first-class proof artifact instead of being only an informal reading of thin wrapper lemmas. |
| `Microvmm/Proof/Linux/Console.lean` | Probe readiness and interactive readiness are monotone once observed. | Later transcript bytes and suffix trimming cannot erase the fact that Linux boot or initrd handoff was already seen. |
| `Microvmm/Proof/Linux/BootPlan.lean` | Fixed GDT and e820 writes have named byte-size and span facts, and the normalized final-write view of the Linux boot plan is pairwise disjoint and in range under explicit placement preconditions. | Early boot layout now has an honest whole-plan theorem over the final guest-visible write windows, even though the raw replay order still patches the copied setup header in place. |
| `Microvmm/Proof/Linux/Platform.lean` | The Linux virtio-pci BAR `processQueue` action, explicit-access MMIO BAR route, and MMIO-exit wrapper refine to the same real virtio-rng executor path used at runtime, recover both the executor's concrete write trace and its realization by the Lean guest-write wrappers, and now also lift the packaged `descriptorValidForDevice` judgment from the post-write PCI transport state through that Linux-facing `processQueue` path. | The Linux-facing proof boundary now reaches the real platform MMIO-exit shell while also carrying the accepted-shape descriptor-validity invariant through the Linux BAR/MMIO wrappers instead of stopping at the transport modules alone. |
| `Microvmm/Proof/Linux/Runtime.lean` | The Linux serial and interactive run loops now package the surviving `processQueue` MMIO branch facts: the named raw boundary, the `handleLinuxMmioExit` witness, console preservation, a recovered virtio-rng executor report realized by the Lean guest-write wrappers, and the same packaged descriptor-validity judgment for the resulting device state. | The proved virtio path now reaches the actual bounded Linux run-loop branches that dispatch guest MMIO exits, while being explicit that the current theorem stops at the successful MMIO branch and raw-boundary witness rather than claiming a whole-loop equality or liveness result. |
| `Microvmm/Proof/Virtio/Core.lean` | Successful post-activation writes to `queue_size`, `queue_ready`, and the queue address fields preserve the live and latched queue state. | Queue immutability is now stated once at the shared virtio core instead of being left implicit in runtime tests. |
| `Microvmm/Proof/Virtio/Cve2026_5747.lean` | The queue-latching invariant survives the post-activation queue rewrites used in the Firecracker CVE-2026-5747 bug class. | The repo now has an explicit proof-facing argument for why this virtio implementation does not accept that mutation-after-activation path. |
| `Microvmm/Proof/Virtio/Mmio.lean` | The decoded MMIO write entry point preserves the post-activation queue-latching and DMA-target invariants for queue-size, queue-ready, and queue-address writes; the same decoded queue-write paths now also preserve the packaged descriptor-validity judgment for the current accepted request shape; its notify branch composes with the real rng executor; the standalone explicit-access and MMIO-exit shells recover that executor report; and the one-step run-loop MMIO theorem packages the surviving result up to `VirtioMmioRawBoundary`. | The standalone virtio proof boundary now reaches the real MMIO loop branch that dispatches KVM MMIO exits rather than stopping at thin queue-write wrappers or a lower shell helper, while also letting downstream transport proofs cite descriptor-validity preservation directly at the decoded MMIO write layer. |
| `Microvmm/Proof/Virtio/Pci.lean` | The decoded PCI BAR/common-config write path preserves the same post-activation queue safety properties, now also preserves the packaged descriptor-validity judgment across the decoded queue-write paths, and its notify branch composes with the real rng executor while keeping the same DMA-target view. | The Linux-facing PCI transport now has proof coverage at the decoded BAR/common-config write layer rather than only at wrapper helpers, including a direct transport-level statement that post-activation queue writes cannot make an observed malformed accepted-shape request become valid. |
| `Microvmm/Proof/Virtio/Rng.lean` | The seeded virtio-rng completion path derives its avail, used, and descriptor DMA targets solely from the active queue, successful activation establishes the current accepted queue shape, a successful pure completion plan fixes one queue, one bounded completion, and one descriptor-length payload before the IO path executes it, the same module packages queue-geometry and descriptor-validity invariants for the accepted request shape plus a device-level equivalence theorem showing preserved live queues cannot reclassify the same observed descriptor request, and it now also packages executor-read and completion-write region theorems stating that the concrete offsets touched by the accepted path stay inside already-validated queue/payload regions. | The virtio-rng proof surface now covers DMA-retargeting resistance, accepted-shape descriptor-validity preservation, and the concrete access-region theorem behind the claim that successful virtio-rng completion stays within validated guest regions, all still scoped honestly to the current accepted single-descriptor shape. |
| `Microvmm/Proof/Virtio/RngExecution.lean` | The live virtio-rng executor consumes the validated completion plan by emitting exactly the pure completion write trace, installing the plan's next PRNG state, only toggling the completion-facing device flags, allowing the erased public executor to recover that same report on success, and proving that the executed trace is realized in order by the Lean `guestWriteU*` wrappers. | Virtio proofs now reach the real executor path used by MMIO and PCI down to the Lean guest-write wrappers, leaving only the raw shim-level write calls trusted below that layer. |

`Microvmm/Proof.lean` re-exports this surface so proof readers can start in one place.

## What is still unproved

- Interpreter refinement: there is not yet an end-to-end theorem connecting the pure boot plan to the guest-memory writes performed by the runtime setup path.
- Linux replay-order normalization: the raw boot-plan replay list still includes the copied setup header followed by later field overrides, so the new whole-plan theorem is stated over the normalized final-write view rather than the original replay order.
- Arithmetic and span lemmas: more reusable facts are still needed around guest-memory windows, ring spans, and address arithmetic.
- Runtime liveness claims: there is not yet a proof that the bounded standalone or Linux run loops reach transcript or interactive markers under stated assumptions; the current loop theorems only cover one-step successful MMIO-branch refinement up to recovered executor reports and the named raw boundary.
- Host-boundary specifications: raw KVM/shim calls, socket, stdio, timer, and other host IO remain trusted operational boundaries rather than verified models, even though the virtio path now reaches the Lean wrapper layer above them.
